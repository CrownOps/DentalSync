## 2. 문제정의의 흐름

### 2.1 출발점

처음에는 다음 문제에서 출발했다.

- 지르코니아 디스크의 높은 폐기율
- 디스크 배치 효율 저하
- 숙련자의 감에 의존하는 네스팅
- 폐지르코니아 재활용 가능성

### 2.2 피벗의 이유

현장 인터뷰에서 반복적으로 확인된 것은 단순 폐기율보다 다음 문제였다.

- 잘못된 파일을 CAM에 넣음
- 파일이 누락됨
- 수정본과 최종본이 섞임
- 케이스와 STL 파일이 잘못 매칭됨
- 임플란트 브랜드/어버트먼트 정보가 틀림
- 쉐이드, 재료, 치식 정보가 누락되거나 모호함
- 오류가 가공 후 발견되어 재제작과 신뢰 저하로 이어짐

### 2.3 현재 문제정의

치과 제조는 CAD/CAM 장비로 디지털화된 것처럼 보이지만, 실제 케이스 운영은 여전히 수작업 확인과 숙련자 경험에 의존한다. 특히 치과에서 전달되는 의뢰서, 스캔/STL 파일, 쉐이드 사진, 임플란트 정보가 서로 다른 채널에 흩어져 있어 기공소는 이를 사람이 맞춰보며 작업한다. 이 과정에서 파일·케이스·재료 조건의 불일치가 발생하고, 이는 오가공, 리메이크, 납기 지연, 치과와의 신뢰 저하로 이어진다.

## 3. 고객 세그먼트

### 3.1 1차 타겟

| 세그먼트 | 특징 | 니즈 |
|---|---|---|
| 디지털 치기공소 | STL/구강스캔 기반 작업 비중이 높음 | 파일 누락·매칭 오류 방지 |
| 중대형 기공소 | 담당자와 장비가 많고 케이스 수가 많음 | 담당자별 작업 관리, 오류 로그, 품질 관리 |
| 밀링센터 | 외부에서 많은 파일을 받아 가공 | 케이스-파일 매칭, 가공 전 검증 |
| 원내기공실 | 치과 내부에서 보철 제작 | 진료정보-가공정보 연결, 납기/품질 관리 |

### 3.2 후순위 타겟

| 세그먼트 | 이유 |
|---|---|
| 소규모 아날로그 기공소 | 디지털 파일 기반 니즈가 약할 수 있음 |
| 1인 아날로그 치과 | 디지털 의뢰 자체의 전환 유인이 낮을 수 있음 |
| 완전 체어사이드 치과 | 글라우드/JustScan 계열과 겹칠 수 있음 |

## 4. 고객 인사이트

### 4.1 기공소 관점

- 지르코니아 배치 효율은 개선 여지가 있지만, 고객이 더 강하게 체감하는 문제는 `잘못 깎음`이다.
- 잘못 깎음은 파일 누락, 중복 파일, 어버트먼트 매칭 오류, 디스크 두께 선택 오류 등에서 발생한다.
- 현재는 담당자가 폴더, 파일명, 수기 체크, 구두 확인으로 막는다.
- 기공소는 새로운 입력 업무가 늘어나는 것을 싫어한다.
- 따라서 제품은 `추가 입력 플랫폼`보다 `기존 워크플로우 위의 자동 검증 레이어`여야 한다.

### 4.2 치과 관점

- 치과는 기존 기공소와의 신뢰를 매우 중요하게 본다.
- 의뢰서 디지털화 자체가 강한 구매 이유가 되지는 않는다.
- 리메이크가 반복되면 거래 단절의 직접 원인이 된다.
- 치과가 보는 가치는 `기공소 내부 오류 감소 → 보철 품질 안정 → 환자 재내원 감소`이다.

### 4.3 사업 관점

- 국내 기공소 시장 자체는 크지 않으므로, 단순 국내 시장만 보면 확장성이 약할 수 있다.
- 그러나 SaaS, 라이선스, PC당 과금, 글로벌 확장으로 해석하면 기회가 커진다.
- 초기에는 무료/저가 파일럿을 통해 레퍼런스를 확보해야 한다.
- 유명 기공소 또는 치과가 인정한 사례가 생겨야 보수적 시장에서 확산 가능성이 높아진다.

## 5. 솔루션 정의

DetalSync는 기존 CAD/CAM, 3Shape, exocad, hyperDENT 등을 대체하지 않는다. 역할은 다음과 같다.

1. 치과에서 들어온 의뢰 정보를 케이스 단위로 정리한다.
2. OCR 또는 수동 입력으로 종이/이미지 의뢰서를 디지털화한다.
3. 스캔/STL 파일을 케이스와 매칭한다.
4. 파일 누락, 중복, 수정본 혼입, 치식 불일치를 감지한다.
5. 재료, 쉐이드, 임플란트/어버트먼트 정보의 누락과 불일치를 경고한다.
6. 담당자별 작업 진행 상태를 기록한다.
7. 오류 로그와 리메이크 사유를 구조화한다.
8. 치과에 설명 가능한 품질 리포트로 확장한다.

## 6. MVP 범위

### 6.1 MVP 핵심 원칙

- 기존 작업 방식을 크게 바꾸지 않는다.
- 입력 부담을 최소화한다.
- 자동 감지 가능한 것부터 시작한다.
- 법적/의료기기 리스크가 큰 영역은 초기 범위에서 제외한다.
- 네스팅 최적화는 후순위 또는 보조 기능으로 둔다.

### 6.2 MVP 1순위 기능

| 우선순위 | 기능 | 이유 |
|---|---|---|
| 1 | 스캔/STL 파일 누락 감지 | 직접적이고 설명하기 쉬운 페인포인트 |
| 2 | 중복 파일/수정본 혼입 감지 | 오가공 방지 가치가 큼 |
| 3 | OCR 의뢰서 등록 및 수정 | 종이 의뢰서가 아직 많기 때문에 현실적 진입점 |
| 4 | 케이스-파일 매칭 조회/수정 | 기공소 내부 운영의 기본 단위 |
| 5 | 오류 로그 조회/리캡 내보내기 | 품질 리포트와 치과 신뢰 회복으로 확장 가능 |

### 6.3 MVP에서 보류할 기능

| 기능 | 보류 이유 |
|---|---|
| 완전 자동 네스팅 최적화 | 기술 난이도와 도입 검증이 더 필요 |
| CAD/CAM 직접 제어 | 장비/소프트웨어 연동 제약이 큼 |
| EMR/FHIR 완전 연동 | 초기 MVP에는 무겁고 병원별 차이가 큼 |
| 의료용 재료 재활용 | 인증, 품질, 법규 리스크가 큼 |
| 로봇팔/장비 자동화 | 장기 비전으로는 가능하나 초기 SaaS 범위를 벗어남 |

## 7. 핵심 사용자 플로우 초안

### 7.1 OCR 의뢰서 기반 플로우

1. 사용자가 의뢰서 이미지를 업로드한다.
2. OCR이 환자/케이스 식별 정보, 치식, 재료, 쉐이드, 납기, 임플란트 정보를 추출한다.
3. 사용자가 OCR 결과를 검토하고 수정한다.
4. 시스템이 케이스를 생성한다.
5. 사용자가 STL/스캔 파일을 업로드하거나 폴더에서 가져온다.
6. 시스템이 의뢰 정보와 파일을 매칭한다.
7. 누락/중복/불일치가 있으면 경고한다.

### 7.2 파일 감지 기반 플로우

1. 기공소 폴더 또는 클라우드 수신함을 연결한다.
2. 새 STL/스캔 파일이 들어오면 시스템이 감지한다.
3. 파일명, 생성 시각, 수정 시각, 파일 개수, 케이스 규칙을 분석한다.
4. 기존 케이스와 매칭한다.
5. 누락, 중복, 수정본 혼입, 케이스 혼입 가능성을 표시한다.
6. 담당자가 확인 후 작업 상태를 변경한다.
7. 오류는 로그로 저장된다.

## 9. 핵심 리스크

| 리스크 | 대응 방향 |
|---|---|
| 기공소가 문제를 비용으로 체감하지 못할 수 있음 | 오류 로그와 리메이크 비용을 수치화 |
| 추가 입력을 귀찮아할 수 있음 | 자동 감지, OCR, 폴더 연동 중심 설계 |
| 치과가 직접 비용을 내지 않을 수 있음 | 1차 고객을 기공소로 설정하고 치과 가치는 리포트로 제공 |
| 경쟁사가 의뢰서 기능을 이미 제공할 수 있음 | 단순 의뢰서가 아니라 가공 전 검증·품질 리포트로 차별화 |
| 장비/API 연동이 어려울 수 있음 | 초기에는 파일/폴더/업로드 기반 우회 방식 채택 |
| 시장이 보수적임 | 유명 기공소 파일럿, 무료 베타, 치과 인정 사례 확보 |

## 10. 다음 검증 질문

1. 실제 기공소에서 월 몇 건의 파일 누락/중복/수정본 혼입이 발생하는가?
