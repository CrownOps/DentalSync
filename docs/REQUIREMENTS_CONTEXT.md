## 1. 요구사항 정의서 작성 원칙

요구사항 정의서는 기능명을 나열하는 문서가 아니라, 각 기능이 `누가`, `왜`, `어떤 조건에서`, `무엇을 할 수 있어야 하는지`를 명확히 적는 문서이다.

각 세부 요구사항은 아래 항목을 포함하는 것을 권장한다.

| 항목 | 설명 |
|---|---|
| 요구사항 ID | 예: `AUTH-001`, `REQ-001`, `FILE-001` |
| 사용자 역할 | All, 관리자, 기공소 소장, 기공소 직원 등 |
| 기능명 | 사용자가 수행하는 행위 중심으로 작성 |
| 설명 | 기능의 목적과 동작 범위 |
| 선행 조건 | 로그인 여부, 권한, 기존 데이터 존재 여부 |
| 기본 흐름 | 정상 동작 순서 |
| 예외 흐름 | 실패/오류/권한 없음/중복 등 |
| 입력값 | 사용자가 입력하거나 시스템이 받는 값 |
| 출력값 | 화면 표시, 저장 데이터, 알림, 파일 등 |
| 검증 기준 | 완료 여부를 판단할 수 있는 기준 |
| 비고 | 정책, 보류사항, 추후 논의사항 |

## 2. 사용자 역할 정의

| 역할 | 설명 | 주요 권한 |
|---|---|---|
| All | 로그인한 전체 사용자 또는 공통 사용자 | 로그아웃, 회원탈퇴 등 공통 기능 |
| 관리자 | 서비스 운영 관리자 | 사용자 목록 조회, 계정 상태 관리, 전체 통계 조회 가능성 |
| 기공소 소장 | 기공소 조직의 관리자 | 직원 초대, 권한 설정, 담당자별 작업 조회, 케이스 관리 |
| 기공소 직원 | 실제 작업 담당자 | 본인 작업 조회, 케이스 확인, 파일 등록/검수 |
| 치과 사용자 | 향후 확장 역할 | 의뢰 등록, 파일 전달, 진행 상태 조회 가능성 |

> 현재 요구사항 정의에서는 `All`, `관리자`, `기공소 소장`, `기공소 직원`을 우선 정의한다. 치과 사용자는 MVP 범위에 따라 후순위로 둘 수 있다.

## 5. 기능별 세부 요구사항 초안

### AUTH-001. 사용자 로그아웃하기

| 항목 | 내용 |
|---|---|
| 사용자 역할 | All |
| 설명 | 로그인한 사용자가 현재 세션을 종료한다. |
| 선행 조건 | 사용자가 로그인 상태여야 한다. |
| 기본 흐름 | 1. 사용자가 로그아웃 버튼 클릭 → 2. 시스템이 세션/토큰 무효화 → 3. 로그인 화면 또는 메인 화면으로 이동 |
| 예외 흐름 | 이미 만료된 세션이면 로그인 화면으로 이동한다. |
| 검증 기준 | 로그아웃 후 인증이 필요한 페이지에 접근할 수 없어야 한다. |

### AUTH-002. 사용자 회원탈퇴하기

| 항목 | 내용 |
|---|---|
| 사용자 역할 | All |
| 설명 | 사용자가 본인 계정을 탈퇴 처리한다. |
| 선행 조건 | 사용자가 로그인 상태여야 한다. |
| 기본 흐름 | 1. 탈퇴 메뉴 진입 → 2. 경고 문구 확인 → 3. 비밀번호 또는 재인증 → 4. 탈퇴 처리 → 5. 로그아웃 |
| 예외 흐름 | 소장 계정이 기공소의 유일한 관리자라면 권한 위임 후 탈퇴하도록 안내한다. |
| 검증 기준 | 탈퇴 계정은 로그인할 수 없고, 기존 데이터는 정책에 따라 비식별/보관 처리된다. |
| 비고 | 실제 삭제인지 소프트 삭제인지 정책 결정 필요. |

## 6. 의뢰서 요구사항 작성 기준

### REQ-001. OCR 기능으로 새 의뢰서 등록하기

| 항목 | 내용 |
|---|---|
| 사용자 역할 | 기공소 소장/직원 |
| 설명 | 사용자가 종이 의뢰서 이미지 또는 PDF를 업로드하면 OCR이 의뢰 정보를 추출해 새 의뢰서 초안을 생성한다. |
| 입력값 | 의뢰서 이미지/PDF, 치과명, 접수일 등 |
| OCR 추출 후보 | 환자/케이스 식별 정보, 치식, 보철 종류, 재료, 쉐이드, 납기, 임플란트 브랜드, 어버트먼트 정보, 요청사항 |
| 기본 흐름 | 1. 파일 업로드 → 2. OCR 실행 → 3. 추출 결과 표시 → 4. 사용자 확인/수정 → 5. 의뢰서 저장 |
| 예외 흐름 | OCR 신뢰도가 낮은 항목은 `확인 필요`로 표시한다. |
| 검증 기준 | 저장 전 사용자가 반드시 핵심 필드를 확인할 수 있어야 한다. |

### REQ-002. 새 의뢰서 OCR 데이터 수정하기

| 항목 | 내용 |
|---|---|
| 사용자 역할 | 기공소 소장/직원 |
| 설명 | OCR로 생성된 의뢰서 초안을 저장하기 전에 사용자가 잘못 인식된 데이터를 수정한다. |
| 선행 조건 | OCR 결과가 존재해야 한다. |
| 기본 흐름 | 1. OCR 결과 화면 확인 → 2. 필드 수정 → 3. 필수값 검증 → 4. 저장 |
| 검증 기준 | 필수값 누락 시 저장할 수 없어야 한다. |

### REQ-003. 기존 의뢰서 수정하기

| 항목 | 내용 |
|---|---|
| 사용자 역할 | 기공소 소장/직원 |
| 설명 | 이미 저장된 의뢰서의 정보를 수정한다. |
| 선행 조건 | 의뢰서가 이미 등록되어 있어야 한다. |
| 기본 흐름 | 1. 의뢰서 상세 조회 → 2. 수정 버튼 클릭 → 3. 내용 변경 → 4. 저장 |
| 예외 흐름 | 이미 가공 완료된 케이스의 핵심 정보 수정 시 변경 이력을 남긴다. |
| 검증 기준 | 수정 전후 이력이 기록되어야 한다. |

## 7. 스캔파일 오류 감지 요구사항 작성 기준

### FILE-001. 스캔파일 누락 감지하기

| 항목 | 내용 |
|---|---|
| 사용자 역할 | 시스템/기공소 직원 |
| 설명 | 의뢰서 또는 케이스 정보상 필요한 스캔/STL 파일이 등록되지 않았을 때 경고한다. |
| 선행 조건 | 케이스와 필수 파일 기준이 존재해야 한다. |
| 기본 흐름 | 1. 케이스 생성 → 2. 필수 파일 기준 확인 → 3. 등록 파일 개수/종류 비교 → 4. 누락 경고 표시 |
| 출력값 | 누락 파일명/파일 유형, 케이스 ID, 경고 상태 |
| 검증 기준 | 누락 상태가 해소되면 경고가 자동 또는 수동으로 해제되어야 한다. |

### FILE-002. 스캔파일 중복 감지하기

| 항목 | 내용 |
|---|---|
| 사용자 역할 | 시스템/기공소 직원 |
| 설명 | 동일 케이스에 같은 파일이 중복 등록되거나 동일한 해시/파일명 패턴이 발견되면 경고한다. |
| 감지 기준 후보 | 파일명, 파일 해시, 생성일, 크기, 메타데이터, 치식 태그 |

## 8. 요구사항 작성 시 주의점

- `삭제` 기능은 실제 삭제인지 비활성화/보관인지 정책을 먼저 정해야 한다.
- 의료/환자 정보가 포함될 수 있으므로 개인정보 처리 정책을 요구사항에 반영해야 한다.
- 기공소 직원 권한은 최소한 `조회`, `등록/수정`, `삭제`, `권한관리`로 나눠야 한다.
- 오류 감지는 처음부터 100% 자동 판정으로 표현하지 말고 `의심`, `경고`, `확인 필요` 상태로 설계하는 것이 현실적이다.
- OCR은 오인식 가능성이 있으므로 `사람의 최종 확인`을 기본 전제로 둔다.
- 파일 감지는 API 연동보다 폴더 업로드/수동 업로드/메일 수신 기반부터 시작하는 것이 MVP에 적합하다.
